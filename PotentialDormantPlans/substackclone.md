• Core services layer:• Auth service
• Content service
• Subscription service

• Infrastructure layer:• API gateway / load balancer

Core domain model

• User• Fields: id, email, password_hash (or SSO), role (writer/reader/admin), created_at, status.

• AuthorProfile• Fields: user_id, handle, display_name, bio, slug, settings (theme, custom domain later).

• Publication (optional if you want multi‑pub per author)• Fields: id, owner_user_id, name, slug, description, settings_json.

• Post• Fields: id, publication_id, author_user_id, title, slug, body_text (Markdown), status (draft/published), visibility (public/paid/unlisted), created_at, published_at, updated_at, canonical_url (optional), tags.

• Issue (optional, for email “editions” grouping posts)• Fields: id, publication_id, subject_line, body_text, status, scheduled_at, sent_at.

• Subscription• Fields: id, publication_id, subscriber_email, user_id (nullable), status (active/cancelled), source (import/form/api), created_at, unsubscribed_at.

• PaymentPlan / SubscriptionPlan (if monetised)• Fields: id, publication_id, price, interval, currency, stripe_product_id, stripe_price_id.

• Payment / Invoice• Fields: id, subscriber_id, amount, currency, status, provider, provider_ref, created_at.

• Event (analytics)• Fields: id, user_id (nullable), publication_id, type (open, click, view, subscribe, unsubscribe), metadata_json, created_at.

• Functions: sign‑up, login, logout, password reset, session/token management.

Subscription service

• Functions:• add subscriber (form, import, API)

Email service

Key workflows

1. Writer onboarding

1. Sign up → create User.
2. Create publication → name, slug, description.
3. Set basic settings → from‑name, reply‑to, default email template.
4. Start first draft → saved as Post with status=draft.


2. Draft → publish → email

1. Draft post in editor (Markdown).
2. Save → Content service stores draft.
3. Publish → status changes to published, slug generated, search index updated.
4. Send to subscribers?• If yes: create Issue (or reuse post), enqueue email job.
• Worker pulls job, renders email, sends via provider to Subscription list.



3. Reader subscription

1. Reader visits publication page or post.
2. Enters email in subscribe form.
3. Subscription service creates Subscription (pending or active).
4. Optional confirmation email → on click, mark active.
5. Future posts/issues sent to this email.
